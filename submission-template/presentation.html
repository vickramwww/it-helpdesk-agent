Key Decisions

Pre-flight filters run in code, before the LLM
Injection detection, legal keyword scanning, and size capping happen as pure Python upstream. The alternative — a check_injection tool — was rejected because ticket content could influence the LLM's decision to call it.
Tradeoff: Less flexible than LLM-based scanning, but code-level rules cannot be argued past.
Action Executor re-validates allowlist after the LLM decides
The LLM does not have final authority on whether an auto-resolve is permitted. The executor re-checks cancellation_eligible, locked, fraud_flag, and the confidence floor in code. A hallucinated or manipulated auto_action cannot slip through.
Tradeoff: Duplicate logic — but the executor is the safety net, not redundancy.
order_lookup and check_customer are separate tools
The agent only pulls customer history when it needs it — e.g., to detect repeat escalation. On a simple WISMO ticket, customer history is never fetched. This limits data exposure on low-risk tickets.
Tradeoff: Slightly more tool calls on complex tickets, but better privacy posture on simple ones.
Routing is deterministic, not LLM-driven
Given (category, priority, flags, order_state, customer_tier), the routing table produces a queue with no LLM involvement. The LLM's suggested route_to is advisory; the executor uses the table. Routing is auditable and unit-testable independently of model behavior.
Tradeoff: Less adaptive to novel cases, but correct and transparent for known ones.
Confidence floor at 0.70
Below 0.70, the agent routes to triage-review regardless of category/priority assignment. No auto-resolves on uncertain classifications — given that auto-resolve has real-world side effects (cancellations, confirmations), we accepted the false-positive escalation rate.
Tradeoff: More tickets to triage-review than strictly necessary. Tune per-category once eval results are available.

Guardrails & Escalation Logic

The Agent Will Never Autonomously

Issue or authorize any refund — route to human only
Cancel an order if fulfillment has started or order is locked
Take any action on a fraud-flagged order or customer
Act on a ticket from an enterprise-tier customer
Act when confidence < 0.70
Close, resolve, or modify a P1 ticket
Respond to a legal complaint, chargeback, or GDPR request
Take action on a ticket containing prompt injection patterns

How We Used Claude Code

Pattern 1
Write the spec first, reference by section number
"Implement the pre-flight filter from architecture.md §2.2. Pure Python, no tools, runs before any LLM call. Flag: size cap, injection patterns, legal keywords, distress signals."
Result: Injection regex matched the exact threat model in §2.2. Flag names matched the routing table in §4. No correction needed.
Pattern 2
State the constraint before the function
"Add executor_validate(). Constraint: the LLM cannot grant itself permission. Re-check allowlist preconditions in code after the LLM returns. Override to route on failure."
Result: Lambda-based allowlist implemented correctly. executor_override log flag added unprompted. One correction: moved confidence check outside the lambdas to apply uniformly.
Pattern 3 — What Claude caught
Claude flagged an inconsistency before writing
"Write the system prompt encoding all priority, category, and guardrail rules from mandate.md..."
Result: Claude flagged that mandate.md used confidence < 0.75 in one section and 0.70 in another, and asked which was authoritative before writing. We resolved to 0.70 everywhere.
Pattern 4 — What Claude proactively improved
Suggested extracting route() as a standalone pure function
Result: Claude recommended splitting routing into its own function (rather than embedding in the executor) so it would be independently unit-testable without running the full agentic loop. We adopted it immediately.

What We'd Do With More Time

Known Gaps

  • Fix false-positive guardrail triggers on clean P1 tickets — tighten system prompt wording to avoid firing guardrail_triggered when no guardrail actually applies
  • Fix wismo priority classification — p3_wismo_not_self_serve is over-escalating to P2; system prompt needs a clearer P3 definition for WISMO without carrier data
  • Fix P4 auto-resolve detection — agent routes wismo_response and resend_confirmation instead of auto-resolving; likely a system prompt ambiguity around when auto-resolve is appropriate
  • Fix carrier-delivered dispute and customer-requests-human guardrail — agent should trigger guardrail_triggered but currently resolves
  • All tool implementations are mock — no live OMS, CRM, or carrier API calls

Next Iterations

  • Iterate on system prompt for the 4 identified gap categories — fix by tightening prompt rules, never by loosening test expectations
  • Tune confidence floor per-category once more eval data exists — fraud/legal warrant higher, simple WISMO could tolerate lower
  • Replace mock tool bodies with live API calls using the same return schemas (no agent changes required)
  • Build human feedback loop — ops team corrections flow back to improve routing accuracy over time
  • Add queue backpressure — if ops-l2 is overloaded, re-route eligible P2s to ops-l1 with a flag
